## コンテキスト

個人スキルを社内で共有・閲覧可能にしつつ、各自が自由に管理できる仕組みを設計したい。
Slack での議論（[元の会話](../references/raw-conversation.md)）を起点に、2つのアプローチを比較検討した。

## 検討した選択肢

### 選択肢 A: Monorepo（開発者別ディレクトリ＋自動マージ）

単一の社内リポジトリに `users/<name>/` でディレクトリを切り、各自が自分の領域を管理する。
自分のディレクトリのみの変更は GitHub Actions で自動マージする。

### 選択肢 B: サブモジュール集約（個人リポジトリ＋親リポジトリ）

各自が個人リポジトリを持ち、親リポジトリが Git サブモジュールとして集約・参照する。

### 比較

| 観点 | Monorepo | サブモジュール |
|---|---|---|
| 導入コスト | **低い** | 中〜高い |
| 新メンバーのオンボーディング | **ディレクトリ作成のみ** | リポジトリ作成＋サブモジュール登録＋権限設定 |
| 他人のスキルを見る体験 | **GitHub Web でブラウズ、grep 一発** | サブモジュール初期化が必要、Web UI では直接閲覧不可 |
| 個人の自由度 | 高い（ディレクトリ内は自由） | **非常に高い**（リポジトリ全体が自分のもの） |
| ポータビリティ | clone があれば内容は残る | **リポジトリごと持ち出せる** |
| Git の複雑さ | **通常の Git 操作のみ** | サブモジュール特有の操作が必要 |
| 自動化の容易さ | **自動マージは簡単に実装可能** | 親リポジトリの参照更新の自動化が別途必要 |
| 全体検索 | **容易（リポジトリ内 grep）** | ローカルでサブモジュール展開後なら可能 |

## 決定

Monorepo 方式（選択肢 A）を採用する。

「他の人のスキルを覗ける」という主目的に対して、GitHub Web UI でそのままブラウズできる体験はサブモジュール方式では実現できない。導入コスト・Git の複雑さ・全体検索のしやすさいずれも Monorepo が優位。

ポータビリティについては、スキルはテキストファイルの集合であり、ローカルに clone があれば内容は手元に残る。個人で外部にも持ち出したい人は各自で dotfiles 等にミラーする運用とする。

## 影響・トレードオフ

- ポータビリティはサブモジュール方式に劣る。ただし実用上の影響は小さい（上記参照）
- 参加者が増えるとリポジトリサイズは増加するが、テキストファイル中心なので実用上は問題にならない
- 社内全員にスキルが見えることになるため、機密性の高い情報を含めない運用ルールが必要
