## コンテキスト

ADR-001 で Monorepo 方式の採用を決定した。本 ADR では Monorepo の設計方針と、チームスキルとの棲み分けを定める。

実装の詳細（ディレクトリ構造、自動マージの GitHub Actions、CODEOWNERS、オンボーディング手順、運用ルール等）は[設計書](../design/monorepo-design.md)に記載する。

## 決定

### 設計方針

1. **個人領域は完全に自由** — 自分のディレクトリ内の構成・内容は各自の裁量に委ねる
2. **共有のハードルを下げる** — PR を出せば自分の領域は自動マージされる。レビュー不要
3. **閲覧体験を最優先** — GitHub Web UI でそのままブラウズできることを重視する
4. **チームスキルとは別管理** — チーム固有のスキルは各開発リポジトリで管理する（本リポジトリの対象外）

### チームスキルとの棲み分け

| 区分 | 管理場所 | レビュー | 対象 |
|---|---|---|---|
| **個人スキル** | 本リポジトリ `users/<name>/` | 自動マージ | 汎用ノウハウ、個人のワークフロー |
| **Starter Kit** | 本リポジトリ `starter-kit/` | レビュー必須 | 推奨スキル集 |
| **チームスキル** | 各開発リポジトリ `.claude/skills/` 等 | チームのレビュー規約に従う | 業務固有・プロジェクト固有のスキル |

### 還元サイクル

個人で試す → 効果を確認 → `users/` に公開 → 有用なものは `starter-kit/` に昇格提案（PR）→ レビュー → マージ → 全員が利用可能

## 影響・トレードオフ

- 個人領域の自由度を優先するため、スキルの品質にばらつきが生じる。現時点ではスキル作成の知見が十分溜まっていないため、品質管理よりも活性化を優先する判断（[ヘンリー社の事例](../references/henry-skill-management.md)も同様の結論）
- Starter Kit の初期コンテンツを誰が用意するかは別途検討が必要
- セキュリティの線引き（汎用ノウハウ vs 業務固有情報）は[設計書](../design/monorepo-design.md)の運用ルールで定義する
